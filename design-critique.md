# Design Critique: GRG Mobile

**Продукт:** Система управления домофонами (Flutter-приложение + веб-админка)  
**Аудитория:** Жители многоквартирных домов (мобайл) + управляющие компании/администраторы (вебпанель)  
**Стадия:** Active development, production-ready features  
**Дата:** 2026-04-18

---

## Общее впечатление

Продукт имеет хорошую визуальную основу — тёмная тема с фиолетовым акцентом, glassmorphism-карточки, Manrope — выглядит современно. Основные UX-паттерны работают. Главная возможность роста — устранить архитектурные UX-проблемы (навигация, пустые состояния, загрузка) и сделать дизайн системно согласованным. Веб-админка ощущается как внутренний инструмент, построенный без дизайн-системы, и требует более серьёзной доработки.

## Flutter-приложение

### Usability

| Находка | Серьёзность | Рекомендация |
|---------|-------------|--------------|
| Только 2 вкладки в навбаре (Домофон + Настройки), хотя есть здания, события, профиль, устройства | 🔴 Критично | Добавить 3–4 вкладки: **Главная / События / Управление / Профиль**. Сейчас пользователь вынужден нырять через несколько экранов для доступа к базовым функциям |
| `ProfileScreen` не используется в навигации — экран существует, но недоступен через UI | 🔴 Критично | Добавить вкладку «Профиль» в `MainShell`. Также экран нарушает всю дизайн-систему (использует `Colors.red` вместо `AppColors.danger`, нет `GlassCard`, нет `AppColors`) |
| Auth-экран отображает сырой `baseUrl` (`widget.config.baseUrl`) мелким текстом | 🟡 Умеренно | Заменить на понятный контекст: «Сервер: название_сервера» или вообще убрать — пользователю не нужно видеть технический адрес |
| В `_BuildingCard` внутренний padding = 4px (`EdgeInsets.all(4)`) — слишком тесно по сравнению со всеми другими карточками (16px) | 🟡 Умеренно | Унифицировать: использовать `const EdgeInsets.all(16)` как в `_buildCard()` |
| Состояние загрузки кнопки «Открыть» в Incoming Call экране показывает `'...'` | 🟡 Умеренно | Использовать `CircularProgressIndicator` с белым цветом, как в кнопке аутентификации |
| События без снимка показывают крошечную точку `Icons.circle, size: 8` — не информативно | 🟢 Незначительно | Заменить на соответствующую иконку типа события: `Icons.door_front_door` для `door_open`, `Icons.phone_incoming` для `incoming_call` и т.д. |

### Визуальная иерархия

- **Что захватывает взгляд в первую очередь:** На HomeScreen — заголовок «Мои здания» и блок «Последние события». Это правильно, но блок событий визуально слабее, чем карточки зданий, хотя должен быть приоритетным для жильца.
- **Поток чтения:** Верх-вниз, линейный. Работает для простого списка, но теряется для сложной информации.
- **Акцент:** Кнопка «Открыть дверь» в `DoorControlBackendScreen` — правильно размещена и достаточно крупная. Однако на экране входящего звонка три кнопки («Сбросить», «Открыть», «Ответить») визуально равнозначны, хотя «Открыть» — самое критичное действие.

**Рекомендация по Incoming Call экрану:** сделать кнопку «Открыть» центральной и большей (80×80px), «Ответить» — справа средней (64×64px), «Сбросить» — маленькой слева. Приоритизация действий по важности.

### Согласованность

| Элемент | Проблема | Рекомендация |
|---------|----------|--------------|
| `GlassCard` не является настоящим glassmorphism | Нет `BackdropFilter / ImageFilter.blur()` — только полупрозрачный цвет и тень | Добавить `BackdropFilter(filter: ImageFilter.blur(sigmaX: 10, sigmaY: 10))` внутрь `GlassCard` для настоящего эффекта матового стекла |
| `ProfileScreen` полностью выбивается из системы | Использует `Colors.red`, обычный `Scaffold` без фона, нет `GlassCard` | Перестилизовать под общую дизайн-систему |
| Шрифт `Manrope` задан через тему, но `incoming_call_screen.dart` использует `Colors.white` напрямую | Разрыв в цветовой системе | Заменить все `Colors.white` на `AppColors.textPrimary` |
| `ElevatedButton` и `FilledButton` используются как взаимозаменяемые | В разных экранах для одинаковых действий — разные компоненты | Определить правило: `ElevatedButton` = основное действие, `OutlinedButton` = вторичное, `TextButton` = третичное |
| Размеры шрифтов захардкожены в каждом виджете (13, 12, 16, 18, 22) | Нет единой шкалы типографики | Создать `AppTextStyles` класс с именованными стилями (`headingL`, `bodyM`, `labelS` и т.д.) |

### Доступность

- **Цветовой контраст:** `AppColors.textSecondary = Color(0xFFDACBF5)` на полупрозрачном фоне `surface = Color(0x661A0B2E)` — контраст может быть ниже WCAG AA (4.5:1). Необходима проверка.
- **Touch targets:** Кнопки с `padding: const EdgeInsets.all(16)` соответствуют рекомендуемым 48×48dp. Иконки-кнопки в таблице устройств (`IconButton`) — ок.
- **Читаемость:** Размер 12px в событиях и subtitle — на границе допустимого для мобайла (рекомендуется минимум 12sp, но лучше 13sp+).
- **Отсутствие Semantics-меток** на кастомных виджетах (GlassCard, _NavItem) — скрин-ридеры не смогут описать интерактивные элементы.

### Что работает хорошо

- Тёмная тема с фиолетовым акцентом — свежо и узнаваемо, уместно для продукта безопасности/умного дома
- Анимация навбара (`AnimatedContainer + AnimatedSwitcher`) — приятная деталь
- `BouncingScrollPhysics` — нативное ощущение на iOS
- Разделение `backendMode` / `standaloneMode` в HomeScreen прозрачно и не ломает UI
- Кнопка «Открыть дверь» — крупная, занимает всю ширину, нельзя промахнуться
- `SlideRoute` для переходов — согласован и профессионально выглядит
- Состояния ошибки и пустые состояния есть везде (не оставлены как null)

### Приоритетные рекомендации

**1. Расширить навигацию до 4 вкладок**
Текущие 2 вкладки скрывают ключевые функции. Предлагаемая структура:
```
[🏠 Главная] [📋 События] [🔐 Управление] [👤 Профиль]
```
Это сразу решает проблему недоступности ProfileScreen и делает журнал событий первоклассным экраном.

**2. Привести ProfileScreen в соответствие с дизайн-системой**
Сейчас это самый несогласованный экран. Обернуть содержимое в `GlassCard`, заменить `Colors.red` → `AppColors.danger`, убрать сырой `Scaffold` → использовать прозрачный фон с `AppTheme`.

**3. Добавить настоящее glassmorphism в GlassCard**
```dart
// В GlassCard:
ClipRRect(
  borderRadius: BorderRadius.circular(16),
  child: BackdropFilter(
    filter: ImageFilter.blur(sigmaX: 10, sigmaY: 10),
    child: Container(
      decoration: BoxDecoration(
        color: AppColors.surface, // уже полупрозрачный
        borderRadius: BorderRadius.circular(16),
        border: Border.all(color: AppColors.border),
      ),
      child: Padding(padding: padding, child: child),
    ),
  ),
)
```

**4. Skeleton-загрузка для списков**
Заменить `CircularProgressIndicator` в списках зданий/устройств/событий на shimmer-скелетоны. Это значительно улучшает воспринимаемую скорость.

**5. Приоритизировать действия на экране входящего звонка**
Кнопка «Открыть дверь» должна быть визуально доминирующей — это самое важное действие на этом экране.

## Веб-Админ-Панель (admin.html)

### Usability

| Находка | Серьёзность | Рекомендация |
|---------|-------------|--------------|
| 8 вкладок в одну строку, которая переносится на узком экране | 🔴 Критично | Заменить горизонтальные вкладки на вертикальный сайдбар с иконками и текстом |
| Формы создания встроены прямо в карточку данных — нет чёткого разделения «просмотр» / «создание» | 🟡 Умеренно | Использовать модальные окна или слайдинг-панель (drawer) для создания/редактирования |
| Текст таблиц — `font-size: 12px` — слишком мелко для административных данных | 🟡 Умеренно | Поднять до 13–14px для таблиц, 14px для форм |
| Форма NVR-каналов содержит 9 полей в одну строку с `flex-wrap` | 🟡 Умеренно | Разбить на 2 колонки с понятными группами («Подключение», «Каналы», «Именование») |
| Inline form для добавления жителя содержит raw `userId` поле — пользователю нужно знать UUID | 🟢 Незначительно | Добавить поиск/автокомплит по имени/email вместо ввода ID вручную |

### Визуальная иерархия

- Заголовок `h1` в хедере (Doorphone Backend) — это название системы, но не очевидно, **что** пользователь делает на этой странице
- Блок «Настройки backend» (healthcheck) находится выше блока данных, хотя для повседневной работы неважен
- Вкладки (`tabs`) сливаются с общим фоном — активная вкладка недостаточно выделена

### Согласованность

| Элемент | Проблема | Рекомендация |
|---------|----------|--------------|
| Шрифт тела — `JetBrains Mono` (monospace) | Моноширинный шрифт снижает читаемость для UI-текста | Использовать `JetBrains Mono` только для кода/данных; для UI перейти на `Inter`, `Manrope` или `IBM Plex Sans` |
| Акцентный цвет `#a78bfa` в вебпанели ≠ `#8A2BE2` в Flutter-приложении | Бренд выглядит несогласованным | Унифицировать: использовать одинаковый оттенок фиолетового |
| `h2` внутри карточек — цвет `var(--muted)` — заголовки секций выглядят неактивными | Неверная иерархия | Сделать заголовки `var(--text)`, а подписи — `var(--muted)` |

### Доступность

- Поля форм не имеют `<label for="">` — только визуальные лейблы. Скрин-ридеры не смогут правильно связать label с input.
- Кнопки в таблицах — только текстом («Открыть», «Изменить», «Удалить»), без `aria-label` с контекстом.
- Фокусные состояния не кастомизированы — браузерный дефолт выглядит неуместно на тёмном фоне.

### Что работает хорошо

- Тёмная тема с тёмным фоном `#0f0f12` — читаемо, не устаёт взгляд
- Role-based вкладки — отличная идея, разным ролям показывать разный набор функций
- `meta` подписи с API-эндпоинтами (например, `GET /api/health`) — полезно для разработчиков
- Сохранение активной вкладки в `sessionStorage` — хорошая деталь

## Единая дизайн-система: что не хватает

Обе части продукта выиграют от единого design token файла. Сейчас:

| Токен | Flutter | Admin HTML |
|-------|---------|------------|
| Акцент | `#8A2BE2` | `#a78bfa` (другой!) |
| Успех | `#3DD598` | `#22c55e` (другой!) |
| Ошибка | `#FF6B6B` | `#ef4444` (другой!) |
| Шрифт UI | Manrope | JetBrains Mono |

**Рекомендация:** Создать `design-tokens.json` с каноническими значениями и генерировать из него и `app_theme.dart`, и CSS-переменные.
