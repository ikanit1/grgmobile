## Backend (NestJS) — PostgreSQL

### Запуск PostgreSQL (Docker)

Из папки `backend`:

```bash
docker compose up -d
```

PostgreSQL будет доступен на `localhost:5432` (db: `doorphone`, user: `postgres`, password: `postgres`).

### Настройка `.env`

В `backend/.env` должно быть:

- `DB_TYPE=postgres`
- `DB_HOST=localhost`
- `DB_PORT=5432`
- `DB_USERNAME=postgres`
- `DB_PASSWORD=postgres`
- `DB_NAME=doorphone`

### Запуск backend

```bash
npm run build
npm run start
```

### Вебхуки и приложение

- В `.env` задайте `WEBHOOK_SECRET` — секрет для приёма событий от панелей (заголовок `X-Webhook-Secret`). URL вебхука: `POST /api/webhooks/intercom/event` (тело: `deviceId`, `apartmentNumber`).
- Сервер слушает на `0.0.0.0`, чтобы к нему могли обращаться телефон в Wi‑Fi и панели в сети.

### Квартиры и жители (админка)

1. Откройте админку: `http://<IP_ПК>:3000/api/admin`, войдите под учёткой с правами (например SUPER_ADMIN).
2. **Квартиры**: вкладка «Квартиры» → выберите здание по ссылке или нажмите «Создать» → укажите здание, номер квартиры, при необходимости этаж и **расширение** (SIP/номер для вызова внутреннего монитора). Либо «Импорт квартир» (файл по зданию) или «Создать диапазон».
3. **Жители**: вкладка «Жители» → выберите здание и квартиру → «Добавить жителя» (email или телефон, роль). Либо «Импорт жителей» по зданию (CSV/Excel). Альтернатива: житель в приложении подаёт заявку на квартиру, админ во вкладке «Заявки» одобряет.

После этого жители смогут открывать дверь из приложения и при настройке вебхука/FCM получать пуш о входящем звонке.

### Akuvox (в т.ч. X912): настройка панели от бэкенда

- **Номера квартир**: в системе они должны совпадать с номерами, которые панель передаёт в вебхуке (`apartmentNumber`), иначе звонок не попадёт жителям.
- **Расширение (extension)**: если у квартиры задано поле «Расширение», при синхронизации на панель отправляется контакт с полем `Phone` = это расширение. Панель сможет звонить на внутренний монитор по этому номеру. **На самом мониторе** в настройках (Account/Extension) должен быть выставлен тот же номер.
- **Синхронизация вручную**: вкладка «Устройства» → выберите панель(и) Akuvox → «Синхронизировать конфиг и квартиры». На панель отправляются: URL вебхуков (события вызова, открытие двери), whitelist и список квартир здания. **R20/R25**: квартиры в Контакты (contact API). **X912, S532, S562**: квартиры в «Каталог » Пользователь» (user API).
- **Автоматическая синхронизация**: после «Создать диапазон» квартир или импорта квартир по зданию бэкенд в фоне запускает синхронизацию конфига на все панели Akuvox этого здания — повторно нажимать кнопку не обязательно.

### Akuvox S562 и панели в одной подсети

- **Одна подсеть**: если панель (например S562) и сервер бэкенда в одной подсети, в админке у устройства укажите **host** = IP панели (например `192.168.10.53`), при необходимости **порт**. В `.env` задайте **BACKEND_BASE_URL** = URL, по которому панель может достучаться до сервера (например `http://192.168.10.100:3000`). Вебхук будет вызываться по адресу `POST {BACKEND_BASE_URL}/api/webhooks/akuvox`. Удалённая настройка выполняется из админки: «Синхронизировать конфиг и квартиры» — скрипт с сервера по HTTP настраивает панель (SIP, URL событий, whitelist, квартиры). При необходимости настройте **PROVISION_WHITELIST_IP** (whitelist на панели для доступа к её API).
- **X912/S532/S562**: справочник квартир выгружается через user API в «Каталог » Пользователь»; при ошибке contact API скрипт автоматически переключается на user API.
- **Видеопоток с панели**: приложение получает RTSP-URL и подключается к панели напрямую. Просмотр видео возможен только если телефон в той же сети, что и панель (тот же Wi‑Fi или VPN к сети дома/офиса). Для просмотра из любой сети в будущем потребуется проксирование потока на бэкенде (RTSP → HLS/HTTP).

### Быстрые команды

- Очистить данные в БД: `node scripts/reset-database.js`
- Выдать SUPER_ADMIN: `node scripts/make-super-admin.js <email>`
