# Настройка сети для доступа бэкенда к домофонам ЖК

Бэкенд ходит на панели по HTTP (открытие двери, live-url и т.д.). Запросы идут **через интернет** на роутер ЖК. Чтобы бэкенд «увидел» панели, на стороне ЖК нужно выполнить одну из двух схем.

---

## 1. Port Forwarding (простой вариант)

На роутере ЖК пробрасываются порты на внутренние IP панелей.

### Шаги

1. Узнать **внешний (белый) IP** роутера ЖК (или выделенный IP от провайдера).
2. На роутере настроить **проброс портов** (Port Forwarding / Virtual Server), например:
   - Внешний порт **8081** → внутренний IP панели 1, порт **80**
   - Внешний порт **8082** → внутренний IP панели 2, порт **80**
   - При необходимости — порты для RTSP (например 5554, 5555 → 554 на панелях).
3. В админке или при добавлении устройства указать:
   - **Host:** внешний IP ЖК (один и тот же для всех панелей этого дома).
   - **HTTP-порт:** 8081 для первой панели, 8082 для второй и т.д.
   - Логин/пароль панели — как на самом устройстве.

Итог: бэкенд обращается к `http://{внешний_IP_ЖК}:8081/fcgi/do?action=openDoor&index=1` (Akuvox) или к соответствующему URL Uniview — трафик попадает на панель 1.

**Плюсы:** быстро настроить.  
**Минусы:** внешние порты открыты в интернет; желательно ограничить доступ по IP (только сервер бэкенда) и использовать смену портов по возможности.

---

## 2. VPN (безопасный вариант)

Бэкенд-сервер поднимает VPN-сервер; роутер в ЖК подключается к нему как клиент. Панели остаются в локальной сети ЖК; бэкенд обращается к ним по **внутренним VPN-адресам**.

### Общая схема

1. На сервере бэкенда поднять VPN-сервер (например OpenVPN или WireGuard).
2. Роутер ЖК — как VPN-клиент: подключается к серверу, получает виртуальную подсеть (например `10.8.0.0/24`).
3. Панели в ЖК имеют статические IP в локальной сети (например `192.168.1.11`, `192.168.1.12`). Через маршруты на роутере или NAT они доступны с VPN-интерфейса (например `10.8.0.5`, `10.8.0.6` — как настроите).
4. В админке в поле **Host** устройства указывать **VPN-адрес** панели (например `10.8.0.5`), порт — как у панели (80 для HTTP, 554 для RTSP при необходимости).

Итог: бэкенд ходит по защищённому каналу в VPN-сеть ЖК и по внутреннему адресу открывает дверь / запрашивает поток.

**Плюсы:** трафик зашифрован, в интернет не светится порты панелей.  
**Минусы:** нужна настройка VPN на роутере и сервере.

---

## Таймауты и протоколы

- Для HTTP-запросов к панелям в Control-модуле задан **таймаут 10 секунд** (запрос через интернет).
- **Akuvox:** открытие двери — GET `http://{host}:{port}/fcgi/do?action=openDoor&index=1`, Basic Auth.
- **Uniview (LAPI):** открытие двери — POST `/LAPI/V1.0/Channels/{id}/OpenDoor`, Digest-авторизация.

---

## Проверка без домофона

Вместо панели можно указать IP своего компьютера и порт, на котором слушает тестовый сервер (например Webhook.site или простой HTTP-сервер на Node). В приложении нажать «Открыть дверь» — во входящих запросах на компьютере должен появиться GET (Akuvox) или POST (Uniview) от бэкенда. Так проверяется, что цепочка «приложение → бэкенд → роутер → устройство» работает.
