# Система Multi-Tenant: роли, иерархия, подключение устройств

Документ описывает архитектуру платформы управления домофонами с точки зрения ролей, иерархии данных и сценариев подключения устройств. Детали API и запуск — в [BACKEND.md](BACKEND.md). Полная спецификация (иерархия, алгоритмы, матрица доступов, чек-листы, сценарий входящего звонка) — в [docs/FULL_SYSTEM_SPEC.md](docs/FULL_SYSTEM_SPEC.md).

---

## 1. Роли и участники (три уровня)

| Уровень | Кто | Роль в системе | Что видит и делает |
|--------|-----|----------------|---------------------|
| **Поставщик платформы** | Ваша компания | **SUPER_ADMIN** | Полный доступ: все организации, ЖК, здания, устройства, пользователи. Создание организаций, настройка тарифов (subscription_plan, max_complexes), техподдержка, мониторинг. |
| **Организация / владелец ЖК** | УК, застройщик, ТСЖ | **ORG_ADMIN** (вся организация) или **COMPLEX_MANAGER** (конкретный ЖК) | Свои организации/ЖК: создание и редактирование комплексов, зданий, квартир; добавление устройств к зданиям; приглашение жителей в квартиры (user_apartments); просмотр событий и отчётов. |
| **Жители** | Жильцы | **RESIDENT** | Только свои квартиры (через user_apartments): список «своих» зданий/подъездов, устройства этих зданий — открытие двери, просмотр видео, события. Регистрация/логин по email или телефону. |

Проверка доступа реализована в **AccessService** (`src/access/access.service.ts`): по роли и привязке к организации/ЖК/зданиям.

---

## 2. Иерархия данных

- **Организация** (УК) → **Жилой комплекс (ЖК)** → **Здание (подъезд)** → **Квартиры**.
- **Устройства** привязаны к **зданию** (building).
- **Пользователь** привязан к организации/ЖК (для ORG_ADMIN/COMPLEX_MANAGER) или к квартирам через **user_apartments** (для RESIDENT).

Ограничения:

- У организации есть **max_complexes** — при создании нового ЖК проверяется, что текущее количество ЖК в организации не превышает этот лимит.

## 3. Подключение устройств

### 3.1 Способы

| Способ | Кто выполняет | Как |
|--------|----------------|-----|
| **Ручное добавление** | ORG_ADMIN / COMPLEX_MANAGER (или SUPER_ADMIN) | **POST /api/buildings/:id/devices** с телом: `name`, `host`, `type` (AKUVOX / UNIVIEW_IPC / UNIVIEW_NVR / OTHER), `role` (DOORPHONE / CAMERA / NVR), опционально `username`, `password`, `httpPort`, `rtspPort`. Доступ только к зданиям, которые пользователь может редактировать. |
| **ONVIF Discovery** | ORG_ADMIN / COMPLEX_MANAGER | 1) Выбрать здание. 2) **POST /api/houses/:buildingId/discover-onvif** — запуск поиска. 3) **GET /api/houses/:buildingId/discover-onvif/result** — результат. 4) **POST /api/houses/:buildingId/devices** — добавить выбранное устройство (host, type, username, password, role). Текущая реализация discovery — заглушка (пустой список); для реального поиска нужна интеграция с ONVIF (например node-onvif). |
| **Массовый импорт** | Опционально в перспективе | Файл (CSV/Excel) или отдельный API/скрипт с привязкой к building_id. |

### 3.2 Рекомендуемый поток для организации (владелец ЖК)

1. Войти под ORG_ADMIN или COMPLEX_MANAGER.
2. Создать/выбрать ЖК → здание (через CRUD организаций/комплексов/зданий).
3. Подключение устройств: ручной ввод (host, тип, учётные данные, привязка к зданию) или после реализации — ONVIF поиск по зданию.
4. После добавления устройство появляется в списке здания; жители с привязанными квартирами в этом здании получают доступ к открытию двери и просмотру через приложение.

---

## 4. Привязка жителей к квартирам

- **GET /api/apartments/:apartmentId/residents** — список привязанных жителей (доступ по зданию квартиры).
- **POST /api/apartments/:apartmentId/residents** — добавить жителя. Тело: `userId?`, `email?`, `phone?` (один из трёх), `role?` (owner / resident / guest), `validUntil?` (дата окончания доступа).
- **DELETE /api/apartments/:apartmentId/residents/:userId** — удалить привязку.

Доступ: только пользователи с правом доступа к зданию этой квартиры (SUPER_ADMIN, ORG_ADMIN для своей организации, COMPLEX_MANAGER для своего ЖК).

## 5. Плацдарм платформы (что реализовано)

### 5.1 Уже было

- Мультитенантная модель: организации, комплексы, здания, квартиры, пользователи, user_apartments, устройства, event_logs.
- Роли и доступ: SUPER_ADMIN, ORG_ADMIN, COMPLEX_MANAGER, RESIDENT; проверка по зданиям в AccessService.
- Авторизация: регистрация/логин по email или телефону, JWT с ролью и привязкой к org/complex.
- API для жителей: список зданий, устройств здания, открытие двери, live-url, события, WebSocket событий.
- Управление устройствами: Akuvox, Uniview LiteAPI (HTTP + WebSocket), учётные данные в БД (с возможностью шифрования).
- Админка и документация: Swagger (`/docs`), админ-панель (`/api/admin`), health (`/api/health`).

### 5.2 Добавлено в рамках плацдарма

- **CRUD организаций**: POST /api/organizations, GET/PATCH /api/organizations/:id (SUPER_ADMIN — создание; доступ по организации для редактирования).
- **CRUD комплексов**: POST /api/complexes (проверка max_complexes), GET/PATCH /api/complexes/:id.
- **CRUD зданий**: POST /api/buildings, GET/PATCH /api/buildings/:id.
- **CRUD квартир**: POST /api/apartments, GET/PATCH /api/apartments/:id, GET /api/apartments/by-building/:buildingId.
- **Ручное добавление устройства**: POST /api/buildings/:id/devices (name, host, type, role, username?, password?, httpPort?, rtspPort?).
- **Привязка жителей к квартирам**: GET/POST/DELETE /api/apartments/:apartmentId/residents (по userId, email или phone).
- **Админка по ролям**: в админ-панели (`/api/admin`) — разделение разделов для SUPER_ADMIN (организации, тарифы, ЖК, здания, пользователи, устройства) и для ORG_ADMIN/COMPLEX_MANAGER (ЖК, здания, квартиры, устройства, жители — привязка к квартирам). RESIDENT видит сообщение об использовании мобильного приложения.
- **Документ системы**: данный SYSTEM_DESIGN.md.

### 5.3 Опционально / в перспективе

- **ONVIF discovery**: замена заглушки на реальный поиск устройств по сети (например node-onvif).
- **Шифрование учётных данных** при ручном добавлении устройства (CredentialsService уже есть).
- **Приглашение по ссылке/коду**: создание временной записи user_apartments с valid_until и переход пользователя в RESIDENT после регистрации по ссылке.

## 6. Порядок внедрения (выполнено)

1. Ручное добавление устройства — POST /api/buildings/:id/devices.
2. CRUD организаций и иерархии — API для организаций, комплексов, зданий, квартир; проверка max_complexes при создании комплекса.
3. Привязка жителей к квартирам — API для управления user_apartments.
4. Админка/портал по ролям — разделение экранов для SUPER_ADMIN и ORG_ADMIN/COMPLEX_MANAGER в админ-панели.
5. Документ системы — SYSTEM_DESIGN.md (этот файл).
6. ONVIF discovery — по необходимости; заглушка оставлена.

После этого плацдарм готов: поставщик настраивает организации и тарифы; организации создают структуру ЖК/здания/квартиры и подключают устройства (вручную и при желании в будущем через ONVIF); жители пользуются приложением по привязанным квартирам.
